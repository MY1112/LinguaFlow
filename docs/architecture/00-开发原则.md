# LinguaFlow 开发原则

> **版本：v1.0**
>
> 本文档是 LinguaFlow 项目的最高开发规范。
>
> 所有开发者（包括人工开发、Codex、ChatGPT、Claude Code、Cursor 等 AI 工具）都必须遵守本规范。

---

# 一、项目理念

LinguaFlow 是一款 **Local AI Desktop Tool（本地 AI 桌面工具）**。

项目追求：

- 简单
- 清晰
- 可维护
- 易扩展

避免为了架构而架构。

任何设计都应优先考虑开发效率和长期维护成本。

---

# 二、开发原则

## 1. Feature First（功能优先）

所有新增功能必须优先创建新的 Feature。

例如：

```
features/

translate/

writing/

summary/

proofread/
```

禁止：

为了新增功能修改其它 Feature 的业务代码。

---

## 2. 单一职责

每个模块只负责一件事情。

例如：

ModelService：

只负责模型。

PromptService：

只负责 Prompt。

HistoryService：

只负责历史记录。

不要出现：

```
AIService
```

这种职责不明确的模块。

---

## 3. 高内聚

一个 Feature 的代码尽量放在一起。

例如：

```
translate/

api.py

service.py

prompt.py

models.py

widgets/
```

不要把 Prompt 放到其它目录。

不要把翻译逻辑放到公共 Service。

---

## 4. 公共能力共享

只有多个 Feature 都会使用的功能，才能放入：

```
services/
```

例如：

- ModelService
- PromptService
- ConfigService
- HistoryService

---

## 5. 第三方统一封装

所有第三方库统一放入：

```
integrations/
```

例如：

- llama.cpp
- PaddleOCR
- 剪贴板
- 全局快捷键
- 截图

Feature 不允许直接调用第三方库。

必须通过统一封装。

---

## 6. Runtime 管理运行状态

所有程序运行期间的状态统一管理。

包括：

- 当前模型
- 模型是否加载
- 后台线程
- 推理队列
- 全局上下文

禁止：

使用全局变量保存运行状态。

---

## 7. UI 只负责展示

UI 不允许：

- 推理
- OCR
- Prompt 拼接
- 配置解析
- 文件读写

UI 只负责：

- 用户交互
- 展示结果
- 事件触发

---

## 8. Prompt 属于 Feature

Prompt 与业务紧密相关。

例如：

```
translate/prompt.py

writing/prompt.py

summary/prompt.py
```

不要建立统一的 Prompt 仓库。

---

## 9. 日志统一管理

禁止：

```
print(...)
```

统一使用 Logger。

日志等级：

- DEBUG
- INFO
- WARNING
- ERROR

---

## 10. 配置统一管理

所有配置统一由 ConfigService 管理。

禁止：

任何模块直接读取配置文件。

---

# 三、目录原则

项目目录：

```
src/

app/

ui/

features/

services/

integrations/

runtime/

core/

resources/
```

新增功能：

优先新增 Feature。

不要新增新的顶层目录。

---

# 四、Feature 规范

每一个 Feature 建议保持一致结构：

```
feature_name/

api.py

service.py

prompt.py

models.py

widgets/

resources/
```

如果某个目录暂时不需要，可以省略。

---

# 五、模块依赖规则

允许：

```
Feature

↓

Services

↓

Integrations
```

禁止：

```
Feature A

↓

Feature B
```

Feature 之间不得直接依赖。

---

允许：

```
UI

↓

Feature
```

禁止：

```
Feature

↓

UI
```

---

# 六、命名规范

目录：

使用小写。

例如：

```
translate

writing

summary
```

Python 文件：

使用 snake_case。

例如：

```
model_service.py

config_service.py
```

类：

使用 PascalCase。

例如：

```
TranslateController

ModelService
```

函数：

使用动词。

例如：

```
translate()

load_model()

save_history()
```

---

# 七、代码规范

建议：

- 一个文件尽量不超过 300 行。
- 一个类尽量不超过 300 行。
- 一个函数尽量不超过 50 行。

超过时，应考虑拆分。

---

# 八、新增功能流程

新增一个功能时，遵循以下流程：

1. 创建 Feature 目录。
2. 创建基础文件。
3. 编写 Prompt。
4. 实现业务逻辑。
5. 接入 UI。
6. 编写测试。
7. 更新文档。

---

# 九、禁止事项

禁止：

- God Object（万能对象）
- God Service（万能 Service）
- 超大文件
- 超大类
- 超长函数
- 循环依赖
- 复制粘贴 Prompt
- 使用全局变量管理状态

---

# 十、项目目标

LinguaFlow 不追求复杂架构。

而是追求：

- 每个 Feature 独立
- 每个模块职责清晰
- 新人容易理解
- AI 容易开发
- 长期维护成本低

这是整个项目始终遵循的核心原则。
