# LinguaFlow 工程架构

> 版本：v1.0
>
> 本文档描述 LinguaFlow 的整体工程组织方式。
>
> 重点说明项目目录、模块职责和调用关系，而不是业务实现细节。

---

# 一、设计目标

LinguaFlow 是一个本地 AI 桌面工具。

工程架构遵循以下目标：

- Feature 优先
- 模块职责清晰
- 降低耦合
- 易于扩展
- 易于维护
- AI 能快速理解项目结构

本项目不追求复杂架构，而是追求简单、稳定、长期可维护。

---

# 二、工程目录

```
src/

├── app/
├── ui/
├── features/
├── services/
├── integrations/
├── runtime/
├── core/
├── resources/
└── main.py
```

各目录职责如下。

---

# 三、app

负责程序生命周期。

例如：

- 程序启动
- 初始化配置
- 初始化日志
- 初始化模型
- 初始化快捷键
- 创建主窗口
- 注册系统托盘
- 程序退出

这里不编写业务逻辑。

---

# 四、ui

负责所有公共界面。

例如：

```
ui/

main_window.py

tray.py

dialogs/

components/

themes/
```

职责：

- 页面展示
- 用户输入
- 事件触发

禁止：

- AI 推理
- OCR
- 配置读写
- Prompt 处理

---

# 五、features

项目核心目录。

每一个业务功能都是一个独立 Feature。

例如：

```
features/

translate/

writing/

summary/

history/

settings/
```

新增功能时：

优先新增 Feature。

不要修改其它 Feature。

---

每个 Feature 建议采用统一结构：

```
feature_name/

manifest.py

api.py

service.py

prompt.py

models.py

widgets/

resources/
```

其中：

api.py

负责：

UI 与业务之间的协调。

service.py

负责：

业务逻辑。

prompt.py

负责：

Prompt 模板。

models.py

负责：

本 Feature 使用的数据对象。

widgets/

负责：

Feature 专属 UI。

resources/

负责：

Feature 专属资源。

---

# 六、services

存放所有 Feature 共用的业务服务。

例如：

```
services/

model_service.py

prompt_service.py

config_service.py

history_service.py
```

原则：

只有多个 Feature 都需要使用的能力才能放入这里。

不要把某个 Feature 的业务放到这里。

---

# 七、integrations

封装所有第三方能力。

例如：

```
integrations/

llama_cpp/

paddle_ocr/

clipboard/

hotkey/

screenshot/
```

职责：

- 封装第三方 API
- 屏蔽底层实现

Feature 不直接调用第三方库。

统一通过 integrations。

---

# 八、runtime

负责程序运行期间的状态。

例如：

```
runtime/

model_manager.py

thread_pool.py

task_manager.py

app_context.py
```

职责：

- 当前模型
- 推理状态
- 后台线程
- 全局运行状态
- 任务调度

Runtime 不负责业务。

---

# 九、core

项目公共基础。

例如：

```
core/

logger.py

constants.py

exceptions.py

event_bus.py

utils.py
```

任何模块都可以使用 Core。

Core 不依赖任何 Feature。

---

# 十、resources

项目公共资源。

例如：

```
resources/

icons/

fonts/

images/

styles/
```

Feature 自己的资源建议放在各自目录。

公共资源放这里。

---

# 十一、模块调用关系

允许：

```
UI
    │
    ▼
Feature
    │
    ▼
Services
    │
    ▼
Integrations
```

Runtime：

提供运行环境。

Core：

提供公共能力。

---

禁止：

```
Feature A
        │
        ▼
Feature B
```

Feature 之间不得直接调用。

---

禁止：

```
UI

↓

Integrations
```

UI 不允许直接调用第三方能力。

---

禁止：

```
Feature

↓

UI
```

业务不能操作界面。

如需更新界面，应通过 Controller 或事件通知。

---

# 十二、新增 Feature

新增功能时：

例如：

新增：

AI 校对。

创建：

```
features/

proofread/
```

然后：

```
api.py

service.py

prompt.py

models.py
```

最后：

接入 UI。

无需修改其它 Feature。

---

# 十三、设计原则

整个项目始终遵循：

- Feature First
- 单一职责
- 高内聚
- 低耦合
- 公共能力共享
- 第三方统一封装
- 配置统一管理
- 日志统一管理
- 运行状态统一管理

除非新增需求，否则不修改工程架构。
